# Fashion Recommendation Engine

**Course:** CMPT 311 — Machine Learning  
**Repository:** https://github.com/lmorgenthaler/mood-recommendation

## Reflective Summary

This project is the Stage-C recommendation scoring engine I built as the final project for CMPT 311 (Machine Learning), and which is also the model that powers the ranking layer of my MōōD senior capstone. The goal is to predict, for a given fashion item and customer context, the likelihood that the item will be *recommended* by that customer — i.e., the binary `Recommended IND` label — so that downstream the model can rank items for a user rather than just returning a raw match. I trained and compared a family of classifiers on the Kaggle Women's Clothing E-Commerce Reviews dataset (≈23,500 reviews across product, customer, and free-text fields): a `DummyClassifier` for an honest baseline, regularized Logistic Regression, a decision tree, Random Forest, and the three modern gradient-boosted families — XGBoost, LightGBM, and CatBoost. Features include the categorical product taxonomy (Division, Department, Class), light customer metadata (Age, Rating), and lightweight text-derived features (review length plus a TextBlob sentiment score and polarity strength), all assembled through a `ColumnTransformer` + `Pipeline` so the same preprocessing is applied at training, validation, and inference time. Model selection used `GridSearchCV`, and models were evaluated end-to-end on accuracy, precision, recall, F1, and ROC-AUC, plus a confusion matrix to understand the error structure. The project's stated explicit purpose was to simulate the future production pipeline: structured attributes coming from boutique catalog metadata (Shopify / Square) and an image-attribute model upstream, with the engine here serving as the scoring layer that ranks items per user.

I chose this project for my portfolio because it was the assignment where the math from class first met an actual product question I cared about. That changed how I worked: I had to think about which model class the *data* justified rather than which one I wanted to use, and I had to be careful about leakage between text-derived sentiment features and a label that is itself partly a sentiment artifact. That mode of reasoning is what the CS goal of software design looks like in an ML context — designing the pipeline so the training/inference split is preserved and the evaluation is comparable across models — and it is also the form the problem-solving goal took here, since the trickiest issues were not bugs but quiet methodological mistakes that only became visible when the confusion matrices did not behave the way I expected. On the Westminster side, the project most directly addresses Critical Thinking (choosing the right baseline, treating the AUC differences between boosted models with appropriate skepticism), Creativity (designing a scoring layer that has a coherent place in a larger recommender architecture, rather than treating the assignment as an isolated classification exercise), and Communication, because the deliverable is a narrative Jupyter notebook that has to walk a reader through the modeling choices in order for the results to be trusted.
