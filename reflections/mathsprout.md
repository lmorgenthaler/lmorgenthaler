# MathSprout

**Course:** CMPT 322 — Software Engineering

## Reflective Summary

MathSprout is an educational web application my CMPT 322 team built to teach first- and second-grade students the early-elementary math fundamentals — digit recognition through pattern matching, then addition, then subtraction — through a set of interactive minigames. The app has two distinct user archetypes that drove most of the architectural decisions: a *student* who plays the games and progresses through skill levels, and a *teacher* who monitors classroom progress through a dashboard. Technically, MathSprout is a containerized full-stack application. The frontend is an Angular (TypeScript / SCSS) single-page app with a modular component-and-service structure, route guards on the authenticated pages, and HTTP interceptors that centralize auth headers and error handling. The backend is a Django REST API using DRF serializers, running under ASGI to allow async and WebSocket-capable endpoints for the live game state. Persistence and authentication run on Supabase (managed Postgres), with serverless edge functions handling the operations that did not need to live inside Django, and the entire stack — frontend, backend, database, static and media files — is wired together through `docker-compose` for reproducible local development and deployment.

I chose MathSprout for my portfolio because it is the project where I most directly exercised the engineering side of software development, as opposed to just the programming side. As a team we owned the full software-engineering loop: scoping the user archetypes, designing the API contract before either side was implemented, splitting the work across frontend and backend members, and integrating across an Angular + Django + Supabase boundary that was nobody's individual home turf. That maps cleanly onto the Computer Science goals of software design (the deliberate separation of game state, progress tracking, and dashboard reporting so each could be developed and tested independently), significant project experience working in a group setting (this is the most multi-person codebase I have contributed to), and problem solving — the integration issues across Angular interceptors, DRF serializers, and Supabase auth tokens were exactly the kind of multi-layer bugs that single-author projects rarely produce. On the Westminster side, the project most strongly addresses Collaboration and Communication, which were the actual bottlenecks on a team this size: keeping a shared mental model of the API, reviewing each other's pull requests, and making sure design decisions were written down where the next person to touch the code could find them. Designing for young learners also pushed Creativity in a useful direction — the constraint that the UI needed to be legible and rewarding to a six-year-old eliminated a lot of choices that would have felt natural for an adult-facing product.
